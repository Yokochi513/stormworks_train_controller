## 1. 概要・目的

Stormworks の列車（複数車両を連結した編成）を、編成内の任意の運転台から統一操作するための**分散型コントローラー**を実装する。各車両は1個のマイコン（= 1 Lua ファイル）を持ち、**隣接車両のコネクタとのみ**コンポジット信号で通信する。中央制御装置は存在せず、向き判定・マスター（運転権）・走行指令・付帯機器制御はすべて**隣へのホップ伝播**で成立させる。

各車両は2つのプロパティ `Has Front Control`（前運転台の有無）と `Has Back Control`（後運転台の有無）によって車種が決まる。運転台を持つ車（前または後の運転台を1つだけ持つ車）のみがマスターになり得る。運転台を持たない車（動力付き客車）は、マスター指令の到来方向から自車の進行方向を判定して走行する。

### ゴール

- 前向き動力車（TA）・後ろ向き動力車（TAB）・動力付き客車（TB）を任意の物理向きで連結し、運転台から編成全体を一貫して走行・制御できる。
- 車両の物理的な向き（前後逆置き）に関係なく、編成全体で進行方向・左右が一致する。
- 車両番号や位置の手動割り当てを一切要さず、コネクタ信号とマスター指令の到来方向から自動で向き・位置・編成端・進行方向を判定する。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **ローカル前 / ローカル後** | 各車両に固定された向き。各車両は前コネクタ・後コネクタを1個ずつ持つ。 |
| **`Has Front Control`（前運転台フラグ）** | その車両が**ローカル前を向いた運転台**を持つかを表すプロパティ。`property.getBool("Has Front Control")`。 |
| **`Has Back Control`（後運転台フラグ）** | その車両が**ローカル後を向いた運転台**を持つかを表すプロパティ。`property.getBool("Has Back Control")`。 |
| **運転台付き車** | 前・後いずれか一方の運転台のみを持つ車（`Has Front Control` と `Has Back Control` の排他的論理和が true）。**マスターになり得る**車は運転台付き車（TA / TAB）のみ。 |
| **TA（前向き動力車）** | `Has Front Control=true, Has Back Control=false`。ローカル前を向いた運転台を持ち、正スロットルでローカル前へ進む。バック OFF 時、編成をグローバル前方へ走らせる。 |
| **TAB（後ろ向き動力車）** | `Has Front Control=false, Has Back Control=true`。ローカル後を向いた運転台を持ち、正スロットルでローカル後へ進む。バック OFF 時、編成をグローバル後方へ走らせる。 |
| **TB（動力付き客車）** | `Has Front Control=false, Has Back Control=false`。運転台を持たず、向きが固有には定まらない動力付き車。マスターになれず、マスター指令の到来コネクタ側を進行方向（マスター側）として走行する。 |
| **マスター** | 編成の運転権を持つ唯一の運転台付き車。編成内のマスター数は常に 0 または 1。 |
| **グローバル前方** | 編成全体で共有される進行方向。正スロットルで編成が進む向き。マスターの運転台が向く向きを基準に定まり、バックで反転する。 |
| **グローバル右 / 左** | グローバル前方を向いたときの右手 / 左手側。ドアの左右はこれを基準とする。 |
| **GFF（Global-Forward-to-Front）** | 「グローバル前方が自車のローカル前を指す」とき true となる各車保持フラグ。スロットル符号・左右反転・編成端判定の基礎。マスター指令の到来コネクタ側と連結整列から導出する。 |
| **編成端 / 前端 / 後端** | 片側のコネクタが未接続の車両が編成端。グローバル前方側の端を前端、反対を後端とする（アクティブなマスターと進行方向に追従して入れ替わる）。 |
| **連結マーカー** | 各車が前後コネクタへ常時出力する、接続検出と前後コネクタ識別のための信号。 |
| **指令パケット** | マスターが生成し編成内を両方向へ伝播させる、走行・付帯・向き（マスター在否と進行方向）情報の集合（コンポジット信号として連結マーカーと同居）。 |

### 車種の 2×2 分類（`Has Front Control` × `Has Back Control`）

`Has Front Control`（前運転台の有無）と `Has Back Control`（後運転台の有無）の組で車種が決まる。

| `Has Front Control` | `Has Back Control` | 呼称 | 説明 | マスター可否 | 動力 |
|---|---|---|---|---|---|
| true | false | **TA（前向き動力車）** | ローカル前向き運転台。正スロットルでローカル前へ。バック OFF でグローバル前方へ進む。 | 可 | あり |
| false | true | **TAB（後ろ向き動力車）** | ローカル後向き運転台。正スロットルでローカル後へ。バック OFF でグローバル後方へ進む。 | 可 | あり |
| false | false | **TB（動力付き客車）** | 運転台なし。向き無関係。マスター指令の到来コネクタ側を進行方向とする。 | 不可 | あり |
| true | true | **（ありえない）** | 両運転台はビルド誤り。**動力なし客車**として扱い、前コネクタ由来情報を後コネクタへ・後コネクタ由来情報を前コネクタへクロス中継するのみ。 | 不可 | なし |

- **マスター可否は「運転台をちょうど 1 つ持つこと」（`Has Front Control` ⊻ `Has Back Control`）で決まる。**
- TA と TAB は運転台の向き（前/後）が異なるだけで、いずれもマスターになり得る運転台付き車である。
- TB は運転台を持たないためマスターになれず、走行・マスター調停指令を中継しつつ、マスター指令の到来方向から自車の進行方向を決める。
- `Has Front Control=true, Has Back Control=true` は本来あり得ない組合せ。検出時は動力を持たない客車とみなし、両コネクタ情報をクロス中継するフェイルセーフ動作とする。

### 編成の構成規則

```
[グローバル前端]                                   [グローバル後端]
  TA ── TB ── TB ── … ── TB ── TAB          （両端運転台）
  TA ── TB ── … ── TB                        （後端は TB：後端は運転台なし）
  TA                                          （単車）
```

- 運転台付き車（TA / TAB）は編成の端にのみ存在する。中間車はすべて運転台なし（TB）。
- 各車両は任意の物理的向きで連結され得る（前後どちらを隣へ向けてもよい）。編成内の相対向きはコネクタ信号から判定し、進行方向はマスター指令の到来方向から判定する。

## 3. 車種別 入出力（運転手↔マイコンの外部 I/O）

コネクタ間（車両間）の信号は §8 のプロトコルで別途定義する。本節はマイコンと運転手操作・車両機器との外部 I/O を示す。

**Bool 入力はすべて push 型**：入力信号は押下している間だけ true になるモーメンタリ（押しボタン）であり、マイコン側で**立ち上がり（OFF→ON）を検出するたびに内部の ON/OFF 状態を反転（トグル）**させる。すなわち「一度押すと ON、もう一度押すと OFF」を Lua 側で実装し、その**内部トグル状態を当該機能の ON/OFF 状態**として扱う。

**Number 入力（スロットル・ブレーキスロットル）は運転台付き車（TA / TAB）のみが持つ。** TB はこれら運転入力を持たない。

### 3.1 共通入力（全車種）

| 信号 | 型 | 備考 |
|---|---|---|
| 右ドア | Bool（push→内部トグル） | 内部状態＝右ドア開閉状態。マスター在時は編成指令、不在時は自車ローカル操作。 |
| 左ドア | Bool（push→内部トグル） | 同上。 |
| 室内ライト | Bool（push→内部トグル） | 内部状態＝室内ライト ON/OFF。 |

### 3.2 運転台付き車（TA / TAB）追加入力

| 信号 | 型 | 値域 | 備考 |
|---|---|---|---|
| スロットル | Number | 0.0〜1.0 | 力行指令（大きさのみ）。符号（前後）は GFF とバックで決まり、出力時に -1.0〜1.0 へ写像。 |
| ブレーキスロットル | Number | 0.0〜1.0 | 常用ブレーキ指令。 |
| スポットライト | Bool（push→内部トグル） | — | 内部状態＝前照灯 ON/OFF（手動）。 |
| リクエスト | Bool（push→内部トグル） | — | 内部状態＝マスター要求。OFF→ON で取得（他方の運転台付き車を解放）、ON→OFF で自車解放（§5.3）。 |
| ブレーキ | Bool（push→内部トグル） | — | 内部状態＝非常/パーキングブレーキ。ON 中ブレーキ 100% 固定。 |
| バック | Bool（push→内部トグル） | — | 内部状態＝進行方向反転（ON=反転、OFF=正方向）。 |

### 3.3 出力（車両機器）

| 出力 | TA | TAB | TB | 備考 |
|---|---|---|---|---|
| スロットル | ✔ | ✔ | ✔ | 認可済み力行値（自車符号適用後、値域 **-1.0〜1.0**）。 |
| ブレーキスロットル | ✔ | ✔ | ✔ | 認可済みブレーキ値（0.0〜1.0）。 |
| 右ドア | ✔ | ✔ | ✔ | 自車ローカル右への写像後。 |
| 左ドア | ✔ | ✔ | ✔ | 自車ローカル左への写像後。 |
| 室内ライト | ✔ | ✔ | ✔ | — |
| スポットライト | ✔ | ✔ | — | 前端の運転台付き車のみ点灯。 |
| テールライト | ✔ | ✔ | — | 後端の運転台付き車のみ自動点灯。 |
| 運転モニター（3×1） | ✔ | ✔ | — | 運転データ表示（§7）。 |

> `Has Front Control=true, Has Back Control=true`（ありえない組合せ）の車は動力なし客車として扱い、スロットル/ブレーキ出力を持たず、両コネクタ情報のクロス中継のみを行う（§5.1 REQ-TYPE-04）。

## 4. システムアーキテクチャ（分析）

### 4.1 制約から導かれる原理

- **隣接通信のみ**：各マイコンは隣接車両としか直接通信できない（`要件メモ` L29）。よってマスター状態・進行方向・走行/付帯指令は隣へのホップ伝播で編成端まで届ける。
- **前後コネクタの識別**：コンポジット信号の内容により、信号が前コネクタ由来か後コネクタ由来かを区別できる（`要件メモ` L28）。これを向き・整列・進行方向判定の基礎とする。
- **進行方向はマスター指令の到来方向から判定**：運転台付き車（TA / TAB）が「自分がマスターである／マスターはこちら側」という情報を編成内方向（運転台と反対側）へ伝播し、各車は受信したコネクタ側を進行方向（マスター側）として解釈する（`仕様変更0608.md`）。
- **手動割り当て禁止**：番号・位置・向きは一切手動設定せず、信号から自動判定する（`要件メモ` L24）。プロパティで与えるのは `Has Front Control` / `Has Back Control` のみ。
- **push 型入力**：Bool 入力はモーメンタリで、内部トグル状態は Lua のアップバリュー/グローバル変数で保持する（コルーチン不可）。リスポーンで内部状態はリセットされる前提。

### 4.2 主要データフロー

1. **連結マーカー伝播（毎ティック・双方向）** → 各車が接続有無・隣接整列・編成端を更新。
2. **push 入力のトグル化** → 各 Bool 入力の立ち上がりを検出し、内部 ON/OFF 状態を更新。
3. **マスター調停** → リクエストの内部トグルでマスター取得/解放。相互排他と過渡クリアを保証。
4. **指令パケット伝播（マスター→両端）** → マスター在否・進行方向・認可スロットル/ブレーキ/ドア側/ライトを各ホップで中継。各車は到来コネクタ側から GFF（進行方向）を導出。
5. **出力写像** → 各車が GFF と編成端情報から自車出力（スロットル符号 -1〜1・左右ドア・前照/尾灯）を決定。
6. **モニタ描画** → 運転台付き車が運転データを 3×1 スクリーンへ描画。

## 5. EARS 要件

> 表記は `.claude/.rules/spec-driven-workflow.md` の EARS リファレンスに従う。各要件はテスト可能・一意・追跡可能であることを要する。

### 5.1 車種・構成（REQ-TYPE）

- **REQ-TYPE-01**：システムは `property.getBool("Has Front Control")` と `property.getBool("Has Back Control")` を読み、両者がちょうど一方のみ true（排他的論理和）の車を運転台付き車（マスターになり得る車）として扱うこと。両者 false の車は動力付き客車（TB、マスター不可）、両者 true の車はありえない組合せとして REQ-TYPE-04 のフェイルセーフ動作とすること。
- **REQ-TYPE-02**：`Has Front Control=true, Has Back Control=false`（TA）の車を、ローカル前向き運転台を持ち正スロットルでローカル前へ進む前向き動力車として扱い、バック OFF 時にグローバル前方へ編成を走らせること。`Has Front Control=false, Has Back Control=true`（TAB）の車を、ローカル後向き運転台を持ち正スロットルでローカル後へ進む後ろ向き動力車として扱い、バック OFF 時にグローバル後方へ編成を走らせること。
- **REQ-TYPE-03**：`Has Front Control=false, Has Back Control=false`（TB）の車を、運転台を持たずマスターになれない動力付き客車として扱い、マスター指令の到来コネクタ側を進行方向（マスター側）として走行・中継させること（§5.2 REQ-ORI-04）。
- **REQ-TYPE-04**：`Has Front Control=true, Has Back Control=true`（あり得ない組合せ）を検出した場合、システムは当該車を動力を持たない客車とみなし、スロットル/ブレーキ出力を行わず、前コネクタ由来情報を後コネクタへ・後コネクタ由来情報を前コネクタへクロス中継するフェイルセーフ動作とすること。マスターにはならないこと。
- **REQ-TYPE-05**：運転台を持たない車（TB および REQ-TYPE-04 のフェイルセーフ車）の場合、システムはリクエスト・バック・スポットライト・スロットル系の運転入力を持たず、走行・マスター調停の指令を中継する役割を担うこと（TB は動力付きのため認可走行値に従い走行する）。

### 5.2 向き・位置・進行方向判定（REQ-ORI）

- **REQ-ORI-01**：システムは各コネクタ（前/後）について、隣接車両の連結マーカー受信有無から、そのコネクタが接続済みか未接続かを判定すること。
- **REQ-ORI-02**：片方のコネクタが未接続である場合、システムは自車を編成端として識別すること。
- **REQ-ORI-03**：接続済みコネクタについて、システムは受信した連結マーカーの内容から、隣接車両が自車と整列（同じ向き）か反転（180°逆）かを判定すること。
- **REQ-ORI-04**：マスターが存在する間、システムは指令パケットが到来したコネクタ側（前コネクタ由来か後コネクタ由来か）からマスターの所在方向を判定し、その到来側を自車の進行方向（GFF）として一意に決定すること。これにより運転台を持たない TB も自車のローカル前/後どちらが進行方向かを定めること。
- **REQ-ORI-05**：システムは車両番号・位置・編成内相対向きの手動設定を一切要求しないこと（コネクタ信号・マスター指令の到来方向からのみ自動判定）。プロパティで与えるのは `Has Front Control` / `Has Back Control` のみ。
- **REQ-ORI-06**：システムは「コネクタ由来の編成内相対向き・整列」と「マスター指令の到来方向」を組み合わせ、自車の物理車体（モーター方向・物理左右ドア・物理前後端）とグローバル方向との対応を一意に決定すること。これにより全車種が、任意の物理向きで連結されても編成全体で整合すること。

### 5.3 マスター・リクエストシステム（REQ-MAS）

- **REQ-MAS-01**：起動直後（リスポーン直後）、システムはいずれの車両もマスターを持たない状態であり、各 Bool 入力の内部トグル状態を OFF で初期化すること。
- **REQ-MAS-02**：運転台付き車でリクエストの内部トグル状態が OFF→ON へ遷移したとき（リクエスト押しボタンの立ち上がりで内部状態が ON になったとき）、システムは自車をマスターに設定すること（取得）。
- **REQ-MAS-03**：運転台付き車でリクエストの内部トグル状態が ON→OFF へ遷移したとき、システムは自車のマスターを解放すること（解放）。これにより同一運転台での再押下が解放操作となる。
- **REQ-MAS-04**：運転台付き車でリクエストによりマスターが取得されたとき、システムは他方の運転台付き車（TA / TAB）のマスターを解放し、編成内マスター数を 0 または 1 に保つこと。
- **REQ-MAS-05**：運転台付き車（TA または TAB）が編成に新たに連結されたことを検出したとき、システムは編成内の既存マスターを解放（オフ）し、過渡の発散を防ぐこと。
- **REQ-MAS-06**：マスター権が変更される（取得・解放・移動）とき、システムは各車の Bool 系内部トグル状態（ドア開閉・室内ライト等）を維持すること。
- **REQ-MAS-07**：マスターが不在の間、システムは走行指令（スロットル・ブレーキ）を一切編成へ出さないこと（走行不可）。
- **REQ-MAS-08**：マスターが不在の間、システムは各車の付帯機器（ドア・室内ライト）を自車のローカル入力（内部トグル状態）でのみローカル操作可能とすること（左右は自車ローカル基準）。
- **REQ-MAS-09**：マスターを横取りされて非マスターになった運転台付き車は、自車リクエストの内部トグル状態が ON のまま保持されていても自動でマスターを再取得しないこと（再取得には OFF→ON の遷移を要する）。これにより編成内マスター数が 0 または 1 に保たれること。
- **REQ-MAS-10**：マスターを失った瞬間（横取り・TAB 連結クリア等）、システムは自車リクエストの内部トグル状態を OFF に戻すこと。push 型のためトグル状態は Lua 内部変数で直接管理でき、ハードウェアのリセット線を要さず、以後の単一押下（OFF→ON）で再取得可能とすること。

### 5.4 進行方向・バック（REQ-DIR）

- **REQ-DIR-01**：マスターが存在する間、システムはグローバル前方を「マスターの運転台が向く向き」（前運転台＝マスターのローカル前 / 後運転台＝マスターのローカル後）として、バック OFF 時に定義すること。これにより TA がマスターのときバック OFF で編成はグローバル前方へ、TAB がマスターのときバック OFF でグローバル後方へ進む。
- **REQ-DIR-02**：マスターのバック内部トグル状態が ON の間、システムはグローバル前方を反転（マスターの運転台が向く向きの逆）として定義すること（OFF で正方向）。
- **REQ-DIR-03**：システムは各車のスロットル出力符号を GFF（マスター指令の到来方向）に基づいて決定し、GFF が false の車両では力行方向を反転させること（編成全体が同一のグローバル方向へ進む）。スロットル出力の値域は **-1.0〜1.0** とする。

### 5.5 走行制御（REQ-RUN）

- **REQ-RUN-01**：マスターが存在する間、システムはマスターのスロットル入力（0.0〜1.0）を認可済み力行値（大きさ）として全車へ伝播すること。
- **REQ-RUN-02**：マスターが存在する間、システムはマスターのブレーキスロットル入力（0.0〜1.0）を認可済み常用ブレーキ値として全車へ伝播すること。
- **REQ-RUN-03**：マスターでブレーキ（Bool）の内部トグル状態が ON の間、システムは編成全体のブレーキ出力を 100%（1.0）に固定し、ブレーキスロットル入力より優先すること。
- **REQ-RUN-04**：各車は受信した認可済み力行値（大きさ 0.0〜1.0）に自車の GFF 符号を適用し、スロットル出力（**-1.0〜1.0**）へ写像すること（REQ-DIR-03 と整合）。
- **REQ-RUN-05**：望ましくない条件として、マスター不在・編成断裂・進行方向未確定のいずれかが成立する場合、システムは当該車のスロットル出力を 0、ブレーキを安全側（既定で 1.0）に設定すること。

### 5.6 ドア（REQ-DOOR）

- **REQ-DOOR-01**：マスターが存在する間、システムは右ドア／左ドア指令を「グローバル右／グローバル左」として解釈し、マスターのドア内部トグル状態を編成全体へ伝播すること。
- **REQ-DOOR-02**：各車は、GFF が true のとき「グローバル右＝自車ローカル右」、GFF が false のとき「グローバル右＝自車ローカル左」として、ドア出力を写像すること（左右反転）。
- **REQ-DOOR-03**：マスターが不在の間、各車は自車のローカル右／左ドア入力（内部トグル状態）で自車のドアのみをローカル操作すること（左右はローカル基準、REQ-MAS-08 と整合）。

### 5.7 ライト（REQ-LIGHT）

- **REQ-LIGHT-01**：マスターが存在する間、システムは室内ライト指令（内部トグル状態）を編成全体へ伝播し、全車の室内ライト出力へ反映すること。マスター不在時は各車ローカル操作（REQ-MAS-08）。
- **REQ-LIGHT-02**：スポットライトはマスターのスポットライト内部トグル状態として扱い、システムはその ON/OFF を編成へ伝播すること。
- **REQ-LIGHT-03**：スポットライト指令が ON の間、システムは**前端の運転台付き車（TA / TAB）**のみスポットライト出力を点灯し、それ以外の車両は点灯しないこと。
- **REQ-LIGHT-04**：マスターが存在する間、システムは**後端の運転台付き車（TA / TAB）**のテールライト出力を自動で点灯すること（手動操作なし）。後端が運転台なし（TB）の場合はテールライトを持たないため点灯しない（§7 E8）。
- **REQ-LIGHT-05**：進行方向が変化したとき（バック操作、またはマスターが他端の運転台へ移ったとき）、システムは前端・後端の役割の入替に追従し、スポットライト（前端）・テールライト（後端）の点灯車を切り替えること。
- **REQ-LIGHT-06**：編成が単車（TA または TAB のみ）の場合、当該車は前端かつ後端であるため、システムはスポットライト指令に従い前照灯を扱い、テールライトは点灯しないこと（前照優先のエッジケース、§7）。

### 5.8 運転モニター（REQ-DISP）

- **REQ-DISP-01**：運転台付き車は接続された 3×1 モニターに、運転に必要なデータ（少なくともスロットル・ブレーキ・進行方向・マスター状態）を描画すること（詳細レイアウトは `docs/spec/ui_design.md`）。
- **REQ-DISP-02**：`onDraw` 内では Composite 関数を呼ばず、描画に必要な値は `onTick` でグローバル変数へ受け渡すこと。

### 5.9 連結・通信プロトコル（REQ-COMM）

- **REQ-COMM-01**：各車は毎ティック、前コネクタ・後コネクタそれぞれへ、接続検出と前後識別が可能な連結マーカーを出力すること。
- **REQ-COMM-02**：各車は受信信号の内容から、それが自車の前コネクタ由来か後コネクタ由来かを区別できること。
- **REQ-COMM-03**：マスターが存在する間、システムは指令パケット（マスター在否・進行方向・認可スロットル/ブレーキ/ブレーキ Bool/ドア側/室内ライト/スポット）を生成し、両隣へ伝播し、各中間車は受信パケットを自車フレームへ写像したうえで反対側コネクタへ中継すること。
- **REQ-COMM-04**：マスター在否・進行方向・編成端の各情報は、編成内の全車へ 1 ティック内（または既定の上限ホップ数内）に到達すること。
- **REQ-COMM-05**：編成が断裂（途中で未接続）したとき、断裂で分離した側はマスター指令を受信できなくなり、REQ-RUN-05 の安全側挙動に入ること。
- **REQ-COMM-06**：`Has Front Control=true, Has Back Control=true`（あり得ない組合せ）の車は、指令パケットを自車で消費せず、前コネクタ由来を後コネクタへ・後コネクタ由来を前コネクタへクロス中継すること（REQ-TYPE-04 のフェイルセーフ）。

### 5.10 エッジケース・異常系（REQ-EDGE）

- **REQ-EDGE-01**：運転台付き車（特に TAB）が新たに連結された瞬間に発生し得る状態の発散（「荒ぶり」）を防ぐため、システムは REQ-MAS-05 のマスター解放を適用し、定常状態へ収束させること。
- **REQ-EDGE-02**：編成内に複数のマスター候補が一時的に同時成立した場合、システムは相互排他（REQ-MAS-04）により最終的にマスター数を 0 または 1 に収束させること。
- **REQ-EDGE-03**：システムは運転台付き車の連結・分離の過渡における挙動を Python 等で検証可能な形で設計すること（`要件メモ` L26、検証は §9 の計画に従う）。

### 5.11 非機能要件（REQ-NFR）

- **REQ-NFR-01**：各マイコンの Lua ソースは 8192 バイト（8 KB）以内に収めること（`test_lua_max_bite.py` の制約）。超過する場合はロジック用・表示用チップへの分割を検討する。
- **REQ-NFR-02**：1 ティックの処理は実行時間制限（最大 1000 ms）内に収め、重いループを避けること。
- **REQ-NFR-03**：システムは `require`/`dofile`/`io`/`os`/ティックまたぎコルーチンを使用しないこと（1 マイコン = 1 ファイル）。
- **REQ-NFR-04**：リスポーンで内部状態がリセットされても、数ティック以内に向き・編成端・マスター状態・進行方向を再確立できること。

## 6. データモデル（保持状態）

各マイコンがティックをまたいで保持する主な状態（命名規則・最終的なグローバル名は設計フェーズ/コーディネーターが確定）：

| 状態 | 意味 |
|---|---|
| `has_front_control`, `has_back_control` | プロパティ由来。車種判定の基。 |
| `front_connected`, `back_connected` | 各コネクタの接続有無。 |
| `front_aligned`, `back_aligned` | 隣接車との整列/反転（コネクタ由来）。 |
| `is_master` | 自車がマスターか。 |
| `gff` | グローバル前方が自車ローカル前を指すか。マスター指令の到来コネクタ側＋連結整列から導出。 |
| `is_front_end`, `is_rear_end` | 編成端判定（グローバル方向基準、進行方向に追従）。 |
| `master_present` | 編成内にマスターが存在するか（伝播由来）。 |
| `cab_just_connected` | 運転台付き車の新規連結の立ち上がり検出（REQ-MAS-05）。 |
| **`tgl_*`（各 Bool 入力の内部トグル状態）** | push 型入力ごとに保持する ON/OFF 状態。右ドア・左ドア・室内ライト（全車）、リクエスト・ブレーキ・バック・スポットライト（運転台付き車）。 |
| **`prev_*`（各 Bool 入力の前ティック生入力値）** | push の立ち上がり（OFF→ON）検出用。全 Bool 入力に対して保持する。 |

> **push 型への変更（`仕様変更0608.md`）**：従来の toggle 型（入力レベル＝状態）から push 型（モーメンタリ）へ変更したため、全 Bool 入力について前ティック生値 `prev_*` と内部トグル状態 `tgl_*` の保持が必要となった。リクエストのマスター取得/解放は `tgl_request` の OFF→ON / ON→OFF 遷移で判定する。

## 8. コネクタ・コンポジット信号プロトコル（設計指針）

> 具体的なチャンネル番号（1〜32）の割り当ては、衝突回避責任を持つ `stormworks-coordinator` が `feature_dispatch_*.md` で確定する。本節は**運ぶべき情報**と前後コネクタ識別方式の指針を定める。

各車両は前コネクタ・後コネクタごとに別個のコンポジット出力を持つ。1 つのコネクタ信号は最低限、以下を含む：

- **連結マーカー**（接続検出 + 送信側が前/後どちらのコネクタから送ったかの識別）。受信側はこれと「自車のどちら側で受信したか」を併せ、整列/反転を判定する。
- **指令パケット**（マスター在時のみ有意）：
  - `master_present`（マスター在否）
  - `auth_throttle`（認可力行 0.0〜1.0、大きさ）
  - `auth_brake`（認可ブレーキ 0.0〜1.0）
  - `emergency_brake`（非常ブレーキ Bool）
  - `door_global_right` / `door_global_left`（グローバル左右ドア指令）
  - `room_light`（室内ライト）
  - `spot_on`（スポットライト指令）
  - `global_forward_orientation`（このホップにおける進行方向：受信車が到来コネクタ側から自車 GFF を導出できる形）

> **前後識別・進行方向の最小実装例（指針、確定は設計フェーズ）**：前コネクタには正、後コネクタには負の符号付きマーカー値を載せるなど、符号/専用チャンネルで送信元コネクタを一意化する。受信値の符号と受信チャンネルから整列/反転を導き、**マスター指令が前/後どちらのコネクタから到来したかを進行方向（GFF）に対応付ける**（`仕様変更0608.md` の「来ているローカル方向を進行方向とする」）。マスター（運転台付き車）は自分が運転台側の反対（編成内方向）へ向け、運転台が向く向きを `global_forward_orientation` として送出する。

## 9. 信頼度評価と実装計画

### 9.1 信頼度スコア

**信頼度：80%（中〜高）**

- 確定済み：車種は `Has Front Control` × `Has Back Control` の2フラグで決まり、運転台をちょうど1つ持つ車（TA / TAB）のみマスター可、TB は動力付き客車でマスター指令到来方向から進行方向を判定。Bool 入力はすべて push 型（Lua で内部トグル化）。スロットル入力 0〜1・出力 -1〜1。マスター調停（取得/解放/相互排他/運転台連結クリア）、走行不可時のローカル付帯操作、灯火方針（前端スポット・後端テール、進行方向追従）、ブレーキ Bool 方針。
- 残課題（設計フェーズで詰める）：コネクタ・コンポジット信号の具体ビット割り当て、マスター指令の到来側から GFF・物理左右・物理前後端を導く式、運転台付き車の連結過渡の収束保証。
- 方針：中〜高信頼度のため、**最もリスクの高い「進行方向伝播（到来側判定）+ 運転台連結過渡（荒ぶり）」を PoC** として先行検証し、成功基準を満たしてから全機能へ拡大する。

### 9.2 PoC 成功基準（フェーズ2 前提）

1. 任意向きで連結した編成（TA-TB-TAB、および中間に反転 TB を含む構成）で、マスター指令の到来側から全車の GFF・物理左右・編成端が安定して一意に定まる（全車種で検証）。
2. TA と TAB を連結・分離するシーケンス、およびマスターを TA↔TAB 間で移すシーケンスで、マスターが 0/1 に常に収束し発散しない（Python シミュレーションで検証、`要件メモ` L26）。
3. バック反転、およびマスター移動で全車のスロットル符号（-1〜1）・ドア左右・前後灯火が一括で整合切替する。
