## 1. 概要・目的

Stormworks の列車（複数車両を連結した編成）を、編成内の任意の運転台（TA または TAB）から統一操作するための**分散型コントローラー**を実装する。各車両は1個のマイコン（= 1 Lua ファイル）を持ち、**隣接車両のコネクタとのみ**コンポジット信号で通信する。中央制御装置は存在せず、向き判定・マスター（運転権）・走行指令・付帯機器制御はすべて**隣へのホップ伝播**で成立させる。

### ゴール

- TA / TB / TAB を任意の向きで `TA`・`TA-TB`・`TA-TB-…-TB`・`TA-TB-…-TAB` の形に連結し、運転台から編成全体を一貫して走行・制御できる。
- 車両の物理的な向き（前後逆置き）に関係なく、編成全体で進行方向・左右が一致する。
- 車両番号や位置の手動割り当てを一切要さず、コネクタ信号から自動で向き・位置・編成端を判定する。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **ローカル前 / ローカル後** | 各車両に固定された向き。スロットル入力が正のとき、車両は**自身のローカル前**方向へ進む基準を持つ。各車両は前コネクタ・後コネクタを1個ずつ持つ。 |
| **TA** | 運転台付き車両。`Is TA = true`, `Is TAB = false`。 |
| **TB** | 運転台なしの客車。`Is TA = false`, `Is TAB = false`。 |
| **TAB** | TA を 180° 回転した運転台付き車両。`Is TA = true`, `Is TAB = true`。 |
| **TA型** | TA または TAB（運転台を持ち、マスターになり得る車両）。`Is TA = true` で判定。 |
| **マスター** | 編成の運転権を持つ唯一の TA型。編成内のマスター数は常に 0 または 1。 |
| **グローバル前方** | 編成全体で共有される進行方向。スロットル正で編成が進む向き。マスターのローカル前を基準に定まり、バックで反転する。 |
| **グローバル右 / 左** | グローバル前方を向いたときの右手 / 左手側。ドアの左右はこれを基準とする。 |
| **GFF（Global-Forward-to-Front）** | 「グローバル前方が自車のローカル前を指す」とき true となる各車保持フラグ。スロットル符号・左右反転・編成端判定の基礎。 |
| **編成端 / 前端 / 後端** | 片側のコネクタが未接続の車両が編成端。グローバル前方側の端を前端、反対を後端とする。 |
| **連結マーカー** | 各車が前後コネクタへ常時出力する、接続検出と前後コネクタ識別のための信号。 |
| **指令パケット** | マスターが生成し編成内を両方向へ伝播させる、走行・付帯・向き情報の集合（コンポジット信号として連結マーカーと同居）。 |

### 編成の構成規則

```
[前端]                                   [後端]
  TA ── TB ── TB ── … ── TB ── TAB          （両端運転台）
  TA ── TB ── … ── TB                        （後端は TB：後端は運転台なし）
  TA                                          （単車）
```

- 運転台（TA型）は編成の端にのみ存在する。中間車はすべて TB。
- 各車両は任意の物理的向きで連結され得る（前後どちらを隣へ向けてもよい）。向きはコネクタ信号から判定する。

## 3. 車種別 入出力（運転手↔マイコンの外部 I/O）

コネクタ間（車両間）の信号は §8 のプロトコルで別途定義する。本節はマイコンと運転手操作・車両機器との外部 I/O を示す。すべて push（押下中 true）の Bool 入力と、TA型のみの Number 入力からなる。

### 3.1 共通入力（全車種）

| 信号 | 型 | 備考 |
|---|---|---|
| 右ドア | Bool（push） | 押下でドア操作。マスター在時は編成指令、不在時は自車ローカル操作。 |
| 左ドア | Bool（push） | 同上。 |
| 室内ライト | Bool（push） | 同上（トグル）。 |

### 3.2 TA型（TA / TAB）追加入力

| 信号 | 型 | 値域 | 備考 |
|---|---|---|---|
| スロットル | Number | 0.0〜1.0 | 力行指令。 |
| ブレーキスロットル | Number | 0.0〜1.0 | 常用ブレーキ指令。 |
| スポットライト | Bool（push） | — | 手動トグル（前照灯）。 |
| リクエスト | Bool（push） | — | マスター取得/解放トグル。 |
| ブレーキ | Bool（push） | — | 非常/パーキングブレーキ トグル。ON 中ブレーキ 100% 固定。 |
| バック | Bool（push） | — | 進行方向反転トグル。 |

### 3.3 出力（車両機器）

| 出力 | TA | TB | TAB | 備考 |
|---|---|---|---|---|
| スロットル | ✔ | ✔ | ✔ | 認可済み力行値（自車符号適用後）。 |
| ブレーキスロットル | ✔ | ✔ | ✔ | 認可済みブレーキ値。 |
| 右ドア | ✔ | ✔ | ✔ | 自車ローカル右への写像後。 |
| 左ドア | ✔ | ✔ | ✔ | 自車ローカル左への写像後。 |
| 室内ライト | ✔ | ✔ | ✔ | — |
| スポットライト | ✔ | — | ✔ | 前端 TA型のみ点灯。 |
| テールライト | ✔ | — | ✔ | 後端 TA型のみ自動点灯。 |
| 運転モニター（3×1） | ✔ | — | ✔ | 運転データ表示（§7）。 |

## 4. システムアーキテクチャ（分析）

### 4.1 制約から導かれる原理

- **隣接通信のみ**：各マイコンは隣接車両としか直接通信できない（`要件メモ` L29）。よってマスター状態・進行方向・走行/付帯指令は隣へのホップ伝播で編成端まで届ける。
- **前後コネクタの識別**：コンポジット信号の内容により、信号が前コネクタ由来か後コネクタ由来かを区別できる（`要件メモ` L28）。これを向き・整列判定の基礎とする。
- **手動割り当て禁止**：番号・位置・向きは一切手動設定せず、信号から自動判定する（`要件メモ` L24）。
- **状態保持**：ティックまたぎの状態はアップバリュー/グローバル変数で保持（コルーチン不可）。リスポーンで内部状態はリセットされる前提。

### 4.2 主要データフロー

1. **連結マーカー伝播（毎ティック・双方向）** → 各車が接続有無・隣接整列・編成端・GFF を更新。
2. **マスター調停** → リクエストトグルでマスター取得/解放。相互排他と TAB 連結時クリアを保証。
3. **指令パケット伝播（マスター→両端）** → 認可スロットル/ブレーキ/ドア側/ライトを各ホップで自車フレームへ写像しつつ中継。
4. **出力写像** → 各車が GFF と編成端情報から自車出力（符号・左右・前照/尾灯）を決定。
5. **モニタ描画** → TA型が運転データを 3×1 スクリーンへ描画。

## 5. EARS 要件

> 表記は `.claude/.rules/spec-driven-workflow.md` の EARS リファレンスに従う。各要件はテスト可能・一意・追跡可能であることを要する。

### 5.1 車種・構成（REQ-TYPE）

- **REQ-TYPE-01**：システムは `property.getBool("Is TA")` が true のとき自車を TA型、false のとき TB として扱うこと。
- **REQ-TYPE-02**：システムは `property.getBool("Is TAB")` が true のとき自車を TAB として扱うこと（このとき `Is TA` も true であること）。
- **REQ-TYPE-03**：TB の場合、システムはリクエスト・バック・スポットライト・スロットル系の運転入力を持たず、走行・マスター調停の指令を中継する役割のみを担うこと。

### 5.2 向き・位置判定（REQ-ORI）

- **REQ-ORI-01**：システムは各コネクタ（前/後）について、隣接車両の連結マーカー受信有無から、そのコネクタが接続済みか未接続かを判定すること。
- **REQ-ORI-02**：片方のコネクタが未接続である場合、システムは自車を編成端として識別すること。
- **REQ-ORI-03**：接続済みコネクタについて、システムは受信した連結マーカーの内容から、隣接車両が自車と整列（同じ向き）か反転（180°逆）かを判定すること。
- **REQ-ORI-04**：マスターが存在する間、システムは指令パケットの伝播を通じて自車の GFF（グローバル前方が自車ローカル前を指すか）を一意に決定すること。
- **REQ-ORI-05**：システムは車両番号・位置・向きの手動設定を一切要求しないこと（コネクタ信号からのみ自動判定）。

### 5.3 マスター・リクエストシステム（REQ-MAS）

- **REQ-MAS-01**：起動直後（リスポーン直後）、システムはいずれの車両もマスターを持たない状態であること。
- **REQ-MAS-02**：TA型でリクエスト入力の立ち上がりを検出したとき、システムは自車をマスターに設定すること（マスター取得トグルの ON 側）。
- **REQ-MAS-03**：自車がマスターのとき TA型でリクエスト入力の立ち上がりを検出した場合、システムは自車のマスターを解放し、編成をマスター不在状態にすること（トグルの OFF 側）。
- **REQ-MAS-04**：TA型でリクエストによりマスターが取得されたとき、システムは他方の TA型（TA↔TAB）のマスターを解放し、編成内マスター数を 0 または 1 に保つこと。
- **REQ-MAS-05**：TAB が編成に新たに連結されたことを検出したとき、システムは TA・TAB 双方のマスターを解放（オフ）すること。
- **REQ-MAS-06**：マスター権が変更される（取得・解放・移動）とき、システムは各車の Bool 系出力状態（ドア開閉・室内ライト等）を維持すること。
- **REQ-MAS-07**：マスターが不在の間、システムは走行指令（スロットル・ブレーキ）を一切編成へ出さないこと（走行不可）。
- **REQ-MAS-08**：マスターが不在の間、システムは各車の付帯機器（ドア・室内ライト）を自車のローカル入力でのみローカル操作可能とすること（左右は自車ローカル基準）。

### 5.4 進行方向・バック（REQ-DIR）

- **REQ-DIR-01**：マスターが存在する間、システムはグローバル前方をマスターのローカル前方向（バック OFF 時）として定義すること。
- **REQ-DIR-02**：マスターでバック入力の立ち上がりを検出したとき、システムはグローバル前方を反転（トグル）すること。
- **REQ-DIR-03**：システムは各車のスロットル出力符号を GFF に基づいて決定し、GFF が false の車両では力行方向を反転させること（編成全体が同一のグローバル方向へ進む）。

### 5.5 走行制御（REQ-RUN）

- **REQ-RUN-01**：マスターが存在する間、システムはマスターのスロットル入力（0.0〜1.0）を認可済み力行値として全車へ伝播すること。
- **REQ-RUN-02**：マスターが存在する間、システムはマスターのブレーキスロットル入力（0.0〜1.0）を認可済み常用ブレーキ値として全車へ伝播すること。
- **REQ-RUN-03**：マスターでブレーキ（Bool）がトグル ON の間、システムは編成全体のブレーキ出力を 100%（1.0）に固定し、ブレーキスロットル入力より優先すること。
- **REQ-RUN-04**：各車は受信した認可済み力行値に自車の GFF 符号を適用してスロットル出力へ写像すること（REQ-DIR-03 と整合）。
- **REQ-RUN-05**：望ましくない条件として、マスター不在・編成断裂・向き未確定のいずれかが成立する場合、システムは当該車のスロットル出力を 0、ブレーキを安全側（既定で 1.0）に設定すること。

### 5.6 ドア（REQ-DOOR）

- **REQ-DOOR-01**：マスターが存在する間、システムは右ドア／左ドア指令を「グローバル右／グローバル左」として解釈し、マスターのドア入力を編成全体へ伝播すること。
- **REQ-DOOR-02**：各車は、GFF が true のとき「グローバル右＝自車ローカル右」、GFF が false のとき「グローバル右＝自車ローカル左」として、ドア出力を写像すること（左右反転）。
- **REQ-DOOR-03**：マスターが不在の間、各車は自車のローカル右／左ドア入力で自車のドアのみをローカル操作すること（左右はローカル基準、REQ-MAS-08 と整合）。

### 5.7 ライト（REQ-LIGHT）

- **REQ-LIGHT-01**：マスターが存在する間、システムは室内ライト指令（トグル）を編成全体へ伝播し、全車の室内ライト出力へ反映すること。マスター不在時は各車ローカル操作（REQ-MAS-08）。
- **REQ-LIGHT-02**：スポットライトはマスターの手動トグルとして扱い、システムはその ON/OFF を編成へ伝播すること。
- **REQ-LIGHT-03**：スポットライト指令が ON の間、システムは**前端の TA型**のみスポットライト出力を点灯し、それ以外の車両は点灯しないこと。
- **REQ-LIGHT-04**：マスターが存在する間、システムは**後端の TA型**のテールライト出力を自動で点灯すること（手動操作なし）。
- **REQ-LIGHT-05**：バックにより進行方向が反転したとき、システムは前端・後端の役割の入替に追従し、スポットライト（前端）・テールライト（後端）の点灯車を切り替えること。
- **REQ-LIGHT-06**：編成が単車（TA のみ）の場合、当該車は前端かつ後端であるため、システムはスポットライト指令に従い前照灯を扱い、テールライトは点灯しないこと（前照優先のエッジケース、§7）。

### 5.8 運転モニター（REQ-DISP）

- **REQ-DISP-01**：TA型は接続された 3×1 モニターに、運転に必要なデータ（少なくともスロットル・ブレーキ・進行方向・マスター状態）を描画すること（詳細レイアウトは `docs/spec/ui_design.md`）。
- **REQ-DISP-02**：`onDraw` 内では Composite 関数を呼ばず、描画に必要な値は `onTick` でグローバル変数へ受け渡すこと。

### 5.9 連結・通信プロトコル（REQ-COMM）

- **REQ-COMM-01**：各車は毎ティック、前コネクタ・後コネクタそれぞれへ、接続検出と前後識別が可能な連結マーカーを出力すること。
- **REQ-COMM-02**：各車は受信信号の内容から、それが自車の前コネクタ由来か後コネクタ由来かを区別できること。
- **REQ-COMM-03**：マスターが存在する間、システムは指令パケット（認可スロットル/ブレーキ/ブレーキ Bool/ドア側/室内ライト/スポット/グローバル方向）を生成し、両隣へ伝播し、各中間車は受信パケットを自車フレームへ写像したうえで反対側コネクタへ中継すること。
- **REQ-COMM-04**：マスター在否・進行方向・編成端の各情報は、編成内の全車へ 1 ティック内（または既定の上限ホップ数内）に到達すること。
- **REQ-COMM-05**：編成が断裂（途中で未接続）したとき、断裂で分離した側はマスター指令を受信できなくなり、REQ-RUN-05 の安全側挙動に入ること。

### 5.10 エッジケース・異常系（REQ-EDGE）

- **REQ-EDGE-01**：TA と TAB が新たに連結された瞬間に発生し得る状態の発散（「荒ぶり」）を防ぐため、システムは REQ-MAS-05 のマスター一括解放を適用し、定常状態へ収束させること。
- **REQ-EDGE-02**：編成内に複数のマスター候補が一時的に同時成立した場合、システムは TA↔TAB 相互排他（REQ-MAS-04）により最終的にマスター数を 0 または 1 に収束させること。
- **REQ-EDGE-03**：システムは TA・TAB の連結・分離の過渡における挙動を Python 等で検証可能な形で設計すること（`要件メモ` L26、検証は §9 の計画に従う）。

### 5.11 非機能要件（REQ-NFR）

- **REQ-NFR-01**：各マイコンの Lua ソースは 8192 バイト（8 KB）以内に収めること（`test_lua_max_bite.py` の制約）。超過する場合はロジック用・表示用チップへの分割を検討する。
- **REQ-NFR-02**：1 ティックの処理は実行時間制限（最大 1000 ms）内に収め、重いループを避けること。
- **REQ-NFR-03**：システムは `require`/`dofile`/`io`/`os`/ティックまたぎコルーチンを使用しないこと（1 マイコン = 1 ファイル）。
- **REQ-NFR-04**：リスポーンで内部状態がリセットされても、数ティック以内に向き・編成端・マスター状態を再確立できること。

## 6. データモデル（保持状態）

各マイコンがティックをまたいで保持する主な状態（命名規則・最終的なグローバル名は設計フェーズ/コーディネーターが確定）：

| 状態 | 意味 |
|---|---|
| `is_ta`, `is_tab` | プロパティ由来の車種フラグ。 |
| `front_connected`, `back_connected` | 各コネクタの接続有無。 |
| `front_aligned`, `back_aligned` | 隣接車との整列/反転。 |
| `is_master` | 自車がマスターか。 |
| `gff` | グローバル前方が自車ローカル前を指すか。 |
| `is_front_end`, `is_rear_end` | 編成端判定。 |
| `prev_request`, `prev_back`, `prev_spot`, `prev_room`, `prev_door_*` | 各 Bool 入力の前ティック値（立ち上がり検出用）。 |
| `master_present` | 編成内にマスターが存在するか（伝播由来）。 |
| `tab_just_connected` | TAB 連結の立ち上がり検出（REQ-MAS-05）。 |

## 7. エッジケースマトリクス

| # | 状況 | 期待挙動 | 関連要件 |
|---|---|---|---|
| E1 | TA と TAB が連結された瞬間 | 両マスターを解放し収束。発散しない。 | REQ-MAS-05, REQ-EDGE-01 |
| E2 | TA・TAB 同時にマスター成立 | 相互排他で 1 台へ収束。 | REQ-MAS-04, REQ-EDGE-02 |
| E3 | マスター不在 | 走行不可。付帯のみ各車ローカル操作（左右ローカル基準）。 | REQ-MAS-07, REQ-MAS-08 |
| E4 | 走行中に編成断裂 | 分離側は安全側（スロットル 0・ブレーキ 1.0）。 | REQ-RUN-05, REQ-COMM-05 |
| E5 | バック押下で方向反転 | 全車スロットル符号反転、前端/後端ライト入替。 | REQ-DIR-02, REQ-LIGHT-05 |
| E6 | 反転して連結された中間車 | GFF=false → スロットル符号・ドア左右を反転して整合。 | REQ-DIR-03, REQ-DOOR-02 |
| E7 | 単車（TA のみ） | 前端=後端。スポットは指令に従う、テールは消灯。 | REQ-LIGHT-06 |

## 8. コネクタ・コンポジット信号プロトコル（設計指針）

> 具体的なチャンネル番号（1〜32）の割り当ては、衝突回避責任を持つ `stormworks-coordinator` が `feature_dispatch_*.md` で確定する。本節は**運ぶべき情報**と前後コネクタ識別方式の指針を定める。

各車両は前コネクタ・後コネクタごとに別個のコンポジット出力を持つ。1 つのコネクタ信号は最低限、以下を含む：

- **連結マーカー**（接続検出 + 送信側が前/後どちらのコネクタから送ったかの識別）。受信側はこれと「自車のどちら側で受信したか」を併せ、整列/反転を判定する。
- **指令パケット**（マスター在時のみ有意）：
  - `master_present`（マスター在否）
  - `auth_throttle`（認可力行 0〜1）
  - `auth_brake`（認可ブレーキ 0〜1）
  - `emergency_brake`（非常ブレーキ Bool）
  - `door_global_right` / `door_global_left`（グローバル左右ドア指令）
  - `room_light`（室内ライト）
  - `spot_on`（スポットライト指令）
  - `global_forward_orientation`（このホップにおけるグローバル前方の向き：受信車が自車 GFF を導出できる形）

> 前後識別の最小実装例（指針、確定は設計フェーズ）：前コネクタには正、後コネクタには負の符号付きマーカー値を載せるなど、符号/専用チャンネルで送信元コネクタを一意化する。受信値の符号と受信チャンネルから整列/反転と GFF を導く。

## 9. 信頼度評価と実装計画

### 9.1 信頼度スコア

**信頼度：80%（中〜高）**

- 確定済み：車種判定、マスター調停（取得/解放/相互排他/TAB 連結クリア）、走行不可時のローカル付帯操作、灯火方針、ブレーキ Bool 方針、操作モデル（リクエスト/バックともトグル）。
- 残課題（設計フェーズで詰める）：コネクタ・コンポジット信号の具体ビット割り当て、整列/GFF 導出の厳密式、TA+TAB 連結過渡の収束保証。
- 方針：中〜高信頼度のため、**最もリスクの高い「向き伝播 + TA/TAB 連結過渡（荒ぶり）」を PoC** として先行検証し、成功基準を満たしてから全機能へ拡大する。

### 9.2 PoC 成功基準（フェーズ2 前提）

1. 任意向きで連結した 3 両（TA-TB-TAB）で、全車の GFF と編成端が安定して一意に定まる。
2. TA と TAB を連結・分離するシーケンスで、マスターが 0/1 に常に収束し発散しない（Python シミュレーションで検証、`要件メモ` L26）。
3. バック反転で全車のスロットル符号・ドア左右・前後灯火が一括で整合切替する。
